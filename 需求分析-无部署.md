# 校园“万事达”互助与众包任务平台 —— 需求分析与理论设计文档

**课程名称**：Web开发技术基础课程设计 [cite: 2, 7]
**授课单位**：北京邮电大学 计算机学院（国家示范性软件学院） [cite: 4, 6]

---

## 1. 项目引言与背景分析
在快节奏的校园生活中，学生群体中存在着大量零碎且具有时效性的服务需求，如代领快递、代购餐食、搬运重物等 [cite: 43]。与此同时，许多学有余力的学生希望利用碎片时间通过互助服务获取奖励 [cite: 43]。本项目旨在搭建一个安全、高效的校园众包互助平台，通过引入状态机逻辑、资金托管机制以及信用评价体系，解决校园服务供需匹配的问题 [cite: 43]。

## 2. 角色定义与权限模型 (RBAC理论应用)
**理论联系分析**：系统采用基于角色的访问控制（RBAC, Role-Based Access Control）模型 [cite: 47]。该模型通过将权限与角色绑定，再将角色赋予用户，实现了系统数据和操作的严格隔离，保障了平台的安全性与业务逻辑的清晰。

系统需支持以下三种核心角色：
* **需求方 (Requester)**：系统的服务买方。核心权限包括发布任务（取快递、代买饭等）、预充值资金以及支付任务报酬 [cite: 48]。
* **接单方 (Helper)**：系统的服务卖方。核心权限包括浏览全站任务池、接单、提交完成凭证以及赚取并提取报酬 [cite: 50, 51]。
* **管理员 (Admin)**：系统的监督与维护者。核心权限包括审核违规任务、处理用户争议/申诉、以及监控平台核心数据 [cite: 52, 53]。

## 3. 核心功能需求与理论映射

### 3.1 任务全生命周期管理
**理论联系分析（有限状态机 FSM）**：任务从发布到结束是一个典型的有限状态机（Finite State Machine）模型 [cite: 57]。状态的流转必须由特定的事件（如“接单”、“确认完成”）严格驱动，且禁止非法跃迁（如从“待接单”直接跃迁至“已完成”）。这在开发中锻炼了高并发业务建模能力 [cite: 43]。

**功能需求**：
任务必须严格遵循以下状态流转，且每一次状态变更都必须记录底层日志 [cite: 58]：
1. **待接单**：需求方发布任务，系统验证资金并冻结后，任务进入此初始状态 [cite: 59]。
2. **进行中**：接单方在任务池锁定任务，触发状态变更，此时系统资金进入“担保”状态 [cite: 60]。
3. **待验收**：接单方在线下完成任务后，在系统内上传完成凭证（图片或文字描述） [cite: 61]。
4. **已完成**：需求方审查凭证无误后确认支付，系统执行资金划拨流水 [cite: 62]。
5. **争议中**：若双方对服务结果不满意（如物品损坏、未送达），任意一方可申请挂起任务，由管理员介入裁决 [cite: 63]。

### 3.2 虚拟账户与资金系统
**理论联系分析（事务一致性与 ACID 特性）**：资金系统是平台信任的基石，面临的核心技术挑战是数据库的事务一致性（Transaction Consistency） [cite: 68]。在并发场景下（如同一用户同时发布多个任务，或多人同时抢一个任务），必须保证账户余额的扣减、冻结与任务状态的变更是原子操作，防止“超发”或“资损”。

**功能需求**：
* **余额系统**：每个注册用户拥有独立的虚拟资金账户 [cite: 69]。
* **资金冻结机制**：需求方发布任务时，系统必须校验余额，并将对应金额转入“冻结/锁定”状态，严防“余额不足却成功发布”的恶意行为 [cite: 71]。
* **结算逻辑**：任务合法完成（流转至“已完成”状态）后，系统自动将冻结资金解冻并转入接单方账户 [cite: 72]。
* **可追溯流水记录**：所有的收入、支出、冻结、解冻操作，都必须生成具有唯一单号的流水单据，做到账目 100% 可追溯 [cite: 73]。

### 3.3 实时通知系统
**理论联系分析（发布-订阅模式与长连接）**：为了提供极佳的用户体验，系统需要打破传统的 HTTP 请求-响应模型（轮询拉取），采用基于 WebSocket 的长连接或消息队列（MQ）的发布-订阅模式，实现服务器向客户端的实时事件推送。

**功能需求**：
* **触发场景**：当任务被接单、任务状态发生任意变更，或收到私聊消息时，相关方（需求方或接单方）必须实时收到提醒 [cite: 78]。
* **技术实现**：使用 WebSocket 进行站内实时弹窗或红点推送，或集成第三方邮件 API 作为离线降级通知方案 [cite: 79]。

### 3.4 信用与评价体系
**理论联系分析（动态权重模型）**：信用分是维系社区健康度的核心评价指标，采用动态数据模型计算。

**功能需求**：
* **双向评价**：任务彻底结束后，供需双方需进行服务态度与质量的互评 [cite: 81]。
* **信用分计算模型**：每位用户初始信用分为 100 分 [cite: 82]。系统需根据互评星级以及违约记录（如接单后长时间未响应、恶意取消等）进行动态加减分计算 [cite: 82]。

## 4. 前后端功能详细拆解 (实现层设计)

### 4.1 前端核心功能 (Client-Side)
前端的主要职责是页面渲染、用户交互、数据预处理以及与后端的接口通信。
* **页面渲染与路由管理**：根据用户角色动态渲染多角色工作台；实现前端路由拦截（未登录拦截、越权访问拦截）。
* **核心业务模块 UI**：
  * **任务大厅**：展示任务列表，支持分类筛选和状态过滤，实现分页或触底加载以优化性能。
  * **任务发布表单**：实现表单数据的本地预校验（如金额必须大于0，必填项校验等）。
  * **状态驱动型详情页**：根据任务的当前状态（如待接单、待验收）动态渲染对应的操作按钮（如“上传凭证”、“确认支付”）。
  * **个人/财务看板**：可视化展示当前可用余额、信用分动态以及历史任务流水的明细。
* **实时交互与同步**：实现 WebSocket 客户端，监听后端推送的状态变更，通过系统级的 Toast 弹窗或红点提醒相关用户。

### 4.2 后端核心功能 (Server-Side)
后端的职责是守住系统的安全底线，处理并发业务逻辑，并提供稳定的 API 接口。
* **权限与安全管理网关**：处理注册与登录，下发鉴权 Token（如 JWT）；在接口层严格执行 RBAC 权限校验，防范横向与纵向越权访问。
* **核心业务逻辑与状态机引擎**：在数据访问层（DAO）强制校验状态变更路径，拒绝非法的任务状态跃迁；校验操作人身份的合法性。
* **资金与事务管理 (核心难度)**：使用数据库事务（Transaction）包裹资金流转操作。发布任务时执行“账户校验+资金冻结”的原子操作；任务完成时执行“资金解冻+划拨+生成单据”的一致性操作，彻底杜绝数据脏写。
* **第三方服务与异步调度**：
  * **AI 审核服务**：接收前端传来的任务文本，异步请求大语言模型 API 并将分析结果标签入库。
  * **消息推送服务**：维护 WebSocket 的长连接池，处理心跳保活，并根据业务事件定向分发通知数据包。

## 5. 进阶定制需求：AI 辅助审核模块
*(注：根据项目规划，剔除即时通讯、LBS等复杂功能，深度聚焦 AI 赋能)*

**理论联系分析（AIGC 在业务风控中的应用）**：传统的敏感词过滤往往依赖于静态的正则表达式或前缀树（Trie Tree）算法，容易被变形词绕过且缺乏上下文理解。引入大语言模型（LLM）进行语义级审核，代表了现代 Web 系统风控的演进方向。

**功能需求**：
* **场景**：需求方提交任务发布表单时。
* **机制**：后端异步调用大语言模型 API，对任务的“标题”和“描述”文本进行合规性语义分析 [cite: 89]。
* **交互设计**：
  * AI 模型不直接掌握“拦截/封禁”的生杀大权（避免误伤）。
  * 平台将 AI 的分析结果转化为**文字说明与预警建议**反馈给系统。
  * 例如，若识别到疑似违规或暴力信息，系统会在后台将其标记为“高危需人工复核”，并向管理员展示 AI 给出的风险原因（如：“涉嫌有偿代考，建议拦截”），从而辅助管理决策，提高审核效率 [cite: 89]。

## 6. 开发与过程规范要求
为保证项目工程质量，开发过程需严格遵循以下规范：
1. **协作模式**：2-3人团队协作开发，禁止单干 [cite: 102]。使用云文档进行需求和接口文档的积累 [cite: 104]。
2. **版本控制**：强制使用 Git 进行源代码的版本管理与分支协作 [cite: 105]。
3. **AI 赋能**：在设计与编码阶段，积极引入 AI 编程助手以提升开发效率 [cite: 103]。
4. **技术栈**：开发语言、前端框架及后端架构不限 [cite: 109]，但需具备完整的项目文档（总体设计、详细设计、部署说明及测试报告等） [cite: 108]。
